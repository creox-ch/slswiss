# Contract: Unify Registration

**Спринт:** `unify-registration`
**Дата:** 2026-07-01
**Статус:** `draft` (от Planner, ждёт подтверждения Ksenia)

---

## Цель

Сделать профиль единым независимо от способа входа. Сейчас email-регистрация собирает
кантон/PLZ/согласие, а вход через Google их пропускает → у Google-юзеров профиль неполный
(нет кантона/PLZ, не дано согласие с AGB). Принцип: **вход = только аутентификация**;
недостающие обязательные поля добираются на **общем экране «дозаполни профиль»** после
ЛЮБОГО входа. Источник истины — обязательные поля (имя, email, кантон, PLZ), а не конкретный UI-путь.

## Не входит в спринт

- **Apple OAuth** — реальный вход НЕ подключаем (нужен платный Apple Developer + конфиг в Supabase).
  Кнопка получает честную заглушку `loginSoon('Apple')` («скоро будет доступен»). — Ksenia, 2026-07-01
- **Telegram-вход** — реальный вход = отдельный спринт `telegram-auth`. Пока кнопка ведёт на честную
  заглушку `loginSoon('Telegram')` вместо прежнего фейкового демо-входа (создавал неполный профиль).
- **Интересы** — остаются опциональными и **не персистятся** (как и сейчас в email-регистрации:
  пилюли `int-pill` декоративны, в БД не пишутся). Персист интересов → отдельная задача.
- Изменение полей email-регистрации (шаги rs1/rs2/rs3) — не трогаем.
- Redesign профиля, media-uploads, access-logic.

## Зависимости

- **Зависит от:** `registration-fix` (готов), `legal-consent` (готов — согласие обязательно),
  `cantons-fix` (готов — единый `CANTONS` / `cantonOptionsHtml()`).
- **Блокирует:** `telegram-auth` (переиспользует этот же экран дозаполнения).

- ✅ **Согласие с AGB/Datenschutz** обязательно и на экране дозаполнения после OAuth

## Пользовательские сценарии

### Google-вход без профиля (главный сценарий)
- [ ] Новый юзер жмёт «Google» → проходит OAuth → возвращается на сайт
- [ ] Так как в профиле нет кантона/PLZ → видит экран «Дозаполни профиль» (НЕ попадает сразу на сайт)
- [ ] Экран содержит: кантон (select), PLZ (input), чекбокс согласия с AGB, опц. интересы
- [ ] Заполняет кантон + PLZ + ставит согласие → кнопка «Готово» активна → клик → на главную, залогинен
- [ ] Кантон/PLZ сохранены в `profiles` (и в `user_metadata`) — при перезагрузке профиль полный, экран больше не показывается

### Существующий полный профиль (email или уже дозаполненный)
- [ ] Юзер с уже заполненными кантоном/PLZ входит (email или Google повторно) → экран дозаполнения НЕ показывается → сразу на сайт/профиль
- [ ] Email-регистрация работает как раньше (rs1→rs2→rs3), поведение не изменилось

### Самолечение старых Google-юзеров
- [ ] Юзер, который раньше вошёл через Google и остался без кантона/PLZ, при следующем входе видит экран дозаполнения

### Escape hatch
- [ ] На экране дозаполнения есть ссылка «Выйти» → разлогинивает (не запираем юзера навсегда)

## Edge cases

- [ ] Пустой кантон → кнопка «Готово» disabled
- [ ] PLZ не 4 цифры → ошибка `PLZ — 4 цифры`, кнопка disabled
- [ ] Согласие не отмечено → при клике «Готово» ошибка `Нужно принять условия, чтобы продолжить`, профиль не сохраняется
- [ ] Двойной клик «Готово» не создаёт гонку/двойной апдейт (кнопка уходит в loading)
- [ ] Ошибка сети при сохранении → честное сообщение, экран не закрывается, введённое не теряется
- [ ] Google даёт имя в метаданных → имя не спрашиваем повторно (только кантон/PLZ/согласие)
- [ ] У Google-юзера ещё нет строки в `profiles` (триггер) → сохранение через upsert по `id`, не падает

## UI-инварианты

- [ ] Экран дозаполнения — отдельная модалка `#m-complete` со СВОИМИ id (`#pc-canton`, `#pc-plz`, `#pc-consent`, `#pc-finish`) — без дублей существующих `rs2-*` id
- [ ] Модалку дозаполнения нельзя закрыть по ×/Esc/клику-вне (обязательный шаг); выход — только «Выйти»
- [ ] Кнопка «Готово» disabled пока кантон и PLZ не валидны
- [ ] На клик «Готово» — loading-состояние, повторный клик заблокирован
- [ ] Кантон-select наполняется из единого `CANTONS` (`cantonOptionsHtml()`), 26 кантонов
- [ ] Корректно на экранах ≥320px, без горизонтального скролла

## Технические инварианты

- [ ] После OAuth сессия восстанавливается (`getSession`), профиль грузится, гейт проверяется по `!canton || !plz`
- [ ] Сохранение пишет `canton`, `plz` в `profiles` по `id` и дублирует в `user_metadata` (`auth.updateUser`)
- [ ] Нет ошибок в `console.error` при стандартном flow
- [ ] Существующие проходящие Playwright-тесты (01–19) не сломаны
- [ ] `index.html` не вырос более чем на ~8KB
- [ ] `node --check` на главном script — зелёный

## Решения и заметки

- **2026-07-01 (Planner):** экран дозаполнения — отдельная модалка `#m-complete`, а не переиспользование
  `#m-reg`/rs2, чтобы (а) не плодить дубли id (правило проекта), (б) не тянуть логику email-регистрации
  (`finishRegistration → sbSignUp`). Кантон-список берём из общего `CANTONS`.
- **2026-07-01 (Planner):** интересы на экране показываем опционально, но НЕ персистим — паритет с текущей
  email-регистрацией (там они тоже не пишутся). Персист интересов вынесен из scope.
- **2026-07-01 (Planner):** гейт срабатывает не только после OAuth, но и при любом восстановлении сессии с
  пустым кантон/PLZ → чинит старых Google-юзеров без миграции.
- **⏳ E2E-оговорка:** реальный Google OAuth в CI не прогнать (нужен живой Google-аккаунт + консент-экран).
  Тесты проверяют: (1) полный email-юзер НЕ гейтится; (2) модалка `#m-complete` и её валидация/сохранение
  через программный вызов гейта (`page.evaluate(openProfileCompletion)` на тест-профиле без кантона).
