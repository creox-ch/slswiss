# Contract: Роль «Модератор» — все пути

**Спринт:** `moderator-role` (зонтичный — покрывает уже сделанное + будущие пути одной ролью)
**Дата:** 2026-07-01
**Статус:** `draft` — черновик Planner'а. Часть путей уже ✅ реализована (спринт `moderation`), часть — `todo` (будущие спринты).

> Это **образец процесса** «как не упустить детали». Роль разложена в story map, затем каждый путь
> прогнан по 7 вопросам (happy / отказ / границы / пусто / права / текст ошибки / mobile).
> Каждый пункт-чекбокс = одно проверяемое утверждение = один Playwright-тест.

## Цель

Дать одному человеку («модератор» = `profiles.is_admin`) безопасно управлять всем пользовательским
контентом платформы: заявки в каталог (`services`), события (`events`), а в будущем — статьи
(`articles`) и задачи Биржи (`tasks`). Модерация = смена статуса `pending_moderation → approved | rejected`
и обратно (снять с публикации). `approved` автоматически виден публично через RLS.

## Не входит (в роль, но отдельные роли/спринты)

- Иерархия модераторов, разные уровни прав, назначение модераторов из UI — только ручной флаг `is_admin`.
- Полноценный аудит-лог (кто/когда/что решил) — ⏳ отдельный спринт `moderation-audit`.
- Автомодерация / антиспам-фильтры на вход — вне роли.
- Роль «автор» и роль «аноним» — описаны в своих контрактах; здесь только их пересечение с модератором (RLS).

## Зависимости

- **Зависит от:** `forms-mvp-backend` (заявки приходят со `status='pending_moderation'`), `registration-fix` (модератор — залогиненный юзер).
- **Блокирует:** `user-content-moderation` (статьи), `birzha-auth` (модерация задач), уведомления автору о решении.

## ПУТЬ A — Доступ к модерации (гейт)

**Happy:**
- [x] A1. Модератор (`is_admin=true`) после логина видит кнопку «Модерация» `#mod-btn`.
- [x] A2. Клик по `#mod-btn` (или `window.openModeration()`) открывает экран со списком `#mod-list`.

**Права (главное для этой роли):**
- [x] A3. Гость (не залогинен) НЕ видит `#mod-btn` (`toHaveCount(0)`).
- [x] A4. Обычный юзер (не `is_admin`) НЕ видит `#mod-btn`; `openModeration()` показывает «Доступ только для модераторов».
- [x] A5. **RLS:** анонимный/не-админ ключ прямым PATCH к `/services` НЕ меняет статус (статус остаётся `pending_moderation`). Безопасность на уровне БД, не только UI.

**Отказ / границы / пусто / mobile:**
- [ ] ⏳ A6. `_supabase`/`_profile` ещё не инициализированы на момент клика → кнопка не падает, showNotif, повтор после init.
- [ ] ⏳ A7. Экран модерации адаптивен ≥320px (кнопки решений не наезжают).

## ПУТЬ B — Очередь заявок

**Happy:**
- [x] B1. Экран показывает три секции: 🕓 На модерации (`pending_moderation`), опубликовано (`approved`), отклонено (`rejected`).
- [x] B2. В очередь попадают оба типа: заявки `services` (каталог) и `events` (события); у строки `data-table` и `data-id`.

**Пусто / загрузка:**
- [x] B3. Пока грузится — плейсхолдер «Загрузка…».
- [x] B4. Нет заявок → явный текст «Пока нет заявок» (реализовано в `loadModerationList`).

**Границы:**
- [ ] ⏳ B5. Много заявок (>50) → список не тормозит; пагинация/скролл (сейчас без лимита — проверить/добавить).
- [x] B6. Счётчик заявок на `#mod-btn` — текст «🛡 Модерация (N)» (`_updateModBadge`), тест 12.4.

**Будущие типы контента (расширение роли — те же паттерны):**
- [ ] ⏳ B7. Статьи (`articles`): submit → `pending_moderation` → та же очередь (спринт `user-content-moderation`).
- [ ] ⏳ B8. Задачи Биржи (`tasks`): модерация в той же очереди (спринт `birzha-auth`).

## ПУТЬ C — Изучить заявку перед решением

- [x] C1. По каждой строке видны ключевые поля (имя/заголовок + автор/раздел).
- [ ] ⏳ C2. Раскрыть полную заявку (описание, контакты, изображения) перед решением.
- [ ] ⏳ C3. Превью «как это будет выглядеть в каталоге/афише» до approve.

## ПУТЬ D — Принять решение

**Happy:**
- [x] D1. `pending → approved`: кнопка `.mod-approve` (`✓ Одобрить`) → `update status='approved'` → строка уходит из pending, success-уведомление.
- [x] D2. `pending → rejected`: кнопка `.mod-reject` (`✕ Отклонить`) → `status='rejected'`, строка уходит.
- [x] D3. `approved → rejected`: `.mod-unpublish` (`⨯ Снять с публикации`) → `status='rejected'`, исчезает из публичного каталога.

**Отказ / границы (защита от «ложного успеха» — критично):**
- [x] D4. Строка уходит из списка **только после реального ответа БД** (update прошёл), не оптимистично.
- [x] D5. Двойной клик по кнопке решения не шлёт два update / не ломает состояние.
- [x] D6. Ошибка сети во время решения → честная ошибка, строка остаётся, кнопка снова активна (тест 13.6, `route.abort`).
- [x] D7. Loading-состояние на кнопке во время запроса (кнопка disabled + текст «…»).
- [ ] ⏳ D8. Заявку уже обработал другой модератор (stale) → повторное решение не создаёт конфликт, вменяемое сообщение.

**Расширение:**
- [x] D9. `rejected → approved` (вернуть из «Скрытые», `.mod-republish`), тест 13.7.
- [ ] ⏳ D10. Отклонить **с причиной** (текст комментария сохраняется, доступен автору).

## ПУТЬ E — Результат решения (последствия)

- [x] E1. `approved` → запись видна анонимной роли (проходит публичный SELECT по RLS).
- [x] E2. `approved` services → рендерится в каталоге `#biz-grid`; `approved` events → в афише `#afisha-ev` (см. `catalog-display`).
- [x] E3. `rejected` → НЕ видна публично; остаётся у автора со статусом `rejected`.
- [x] E4 (in-app). Автор видит статус своих заявок в профиле — вкладки «Мой бизнес»/«Мои события», бейдж статуса (`#my-biz-list`/`#my-events-list`), тесты 17.1/17.2. Email-уведомление — ⏳ отдельно.

- [x] T2. UI-гейт (`#mod-btn` только при `_profile.is_admin`) — удобство, НЕ единственная защита (защита — RLS).
- [ ] T3. Нет `console.error` при стандартном flow модерации.
- [ ] T4. Не сломаны существующие зелёные тесты (01–11).
- [ ] ⏳ T5. Аудит: каждое решение пишет `moderated_by` + `moderated_at` (спринт `moderation-audit`).
