# Contract: Forms MVP Backend

**Спринт:** `forms-mvp-backend`
**Дата:** 2026-06-22
**Статус:** `draft` (от Planner, ждёт критики Evaluator)

---

## Цель

Формы «Добавить в каталог» и «Добавить событие» сейчас работают как заглушки: показывают «отправлено на модерацию», но данные никуда не пишутся. Подключить их к Supabase, чтобы заявки реально сохранялись со статусом `pending_moderation`. Модерационный UI (одобрение/отклонение) — НЕ в этом спринте.

## Не входит в спринт

- Модерационный интерфейс (одобрить/отклонить заявку) — отдельный спринт
- Авторизация для отправки форм (регистрация на паузе — пишем от анонимной роли)
- Email-уведомления о новой заявке — отдельная задача
- Редактирование/удаление заявок пользователем
- Загрузка изображений в формах (media-uploads — отдельный спринт)
- Отображение одобренных заявок в публичном каталоге/афише (отдельная задача после модерации)
- Биржа и другие формы — только каталог и события

## Зависимости

- **Зависит от:** существующий Supabase-проект (уже есть: `dwcmiommviauwzkhkbki.supabase.co`)
- **Блокирует:** catalog-moderation, events-moderation (им нужны данные в БД)
- **НЕ зависит от:** registration-fix (формы пишут от анонимной роли)

## Схема БД (Supabase)

### Таблица `catalog_submissions`
- [ ] Создана таблица с полями: `id` (uuid, pk), `created_at` (timestamptz default now()), `status` (text default 'pending_moderation'), плюс поля бизнеса (название, категория, кантон, описание, контакты — финальный список из HTML формы)
- [ ] RLS включён
- [ ] Политика: анонимная роль может только `INSERT`, не `SELECT`/`UPDATE`/`DELETE`
- [ ] Политика INSERT принудительно ставит `status = 'pending_moderation'` (аноним не может вставить со статусом 'approved')

### Таблица `event_submissions`
- [ ] Создана таблица с полями: `id` (uuid, pk), `created_at` (timestamptz default now()), `status` (text default 'pending_moderation'), плюс поля события (название, дата, кантон, описание, тип — финальный список из HTML формы)
- [ ] RLS включён
- [ ] Политика: анонимная роль может только `INSERT`
- [ ] Политика INSERT принудительно ставит `status = 'pending_moderation'`

## Пользовательские сценарии — Каталог

### Happy path
- [ ] Юзер открывает форму «Добавить в каталог» (РАБОЧАЯ кнопка — см. примечание про сломанную верхнюю)
- [ ] Заполняет обязательные поля → кнопка submit активна
- [ ] Кликает submit → данные уходят в Supabase `catalog_submissions`
- [ ] Видит success-экран «Заявка отправлена на модерацию» (честный, данные реально сохранены)
- [ ] В Supabase появилась строка со `status = 'pending_moderation'` и корректными полями
- [ ] Форма очищается или закрывается после успеха

### Валидация
- [ ] Пустые обязательные поля → ошибка под полем, submit не уходит
- [ ] Превышен лимит символов в поле → ошибка или обрезка (по решению из HTML)
- [ ] Невалидный формат (email/телефон бизнеса, если есть) → ошибка

---

## Пользовательские сценарии — События

### Happy path
- [ ] Юзер открывает форму «Добавить событие»
- [ ] Заполняет обязательные поля → submit активен
- [ ] Кликает submit → данные уходят в `event_submissions`
- [ ] Видит success-экран «Событие отправлено на модерацию»
- [ ] В Supabase строка со `status = 'pending_moderation'` и корректными полями
- [ ] Форма очищается или закрывается

### Валидация
- [ ] Пустые обязательные поля → ошибка, submit не уходит
- [ ] Дата события в прошлом → ошибка «Дата должна быть в будущем» (если применимо)
- [ ] Превышение лимита символов → обработано

---

## Edge cases

- [ ] Двойной клик на submit не создаёт две записи (кнопка дизейблится на время запроса)
- [ ] Потеря соединения при submit → ошибка «Нет соединения», данные формы сохранены, можно повторить
- [ ] Ошибка 500/RLS от Supabase → понятное сообщение, НЕ ложный success
- [ ] Supabase SDK вернул null при инициализации → retry loop (известный паттерн из архитектуры), форма не падает
- [ ] Очень длинный ввод (вставка 10000 символов) → обрезается или отклоняется, не ломает запрос
- [ ] Спецсимволы и эмодзи в полях → корректно сохраняются (UTF-8)

## UI-инварианты

- [ ] Кнопка submit дизейблится во время запроса (loading state)
- [ ] Success-экран — НЕ ложный: показывается только после реального подтверждения от Supabase
- [ ] При ошибке success-экран НЕ показывается
- [ ] Сообщение об успехе содержит понятный текст про модерацию (юзер понимает что заявка не опубликована мгновенно)
- [ ] На экранах ≥320px формы не ломают горизонтальный скролл
- [ ] ⚠ ИЗВЕСТНЫЙ БАГ: верхняя кнопка «+ Добавить в каталог» в баннере не открывает форму (из HANDOFF.md). Решить: починить ИЛИ убрать, чтобы не было двух кнопок с разным поведением
