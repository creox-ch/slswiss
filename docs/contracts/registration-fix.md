# Contract: Registration Fix

**Спринт:** `registration-fix`
**Дата:** 2026-06-21
**Статус:** `draft` (от Planner, ждёт критики Evaluator)

---

## Цель

Починить регистрацию пользователей, чтобы к ориентиру ~01.07.2026 можно было открыть запись для тестовой группы 300 человек (дата — ориентир, не жёсткий дедлайн). Сейчас система показывает «email отправлен», но письмо не доходит — это блокирует запуск.

## Не входит в спринт

- Капча (выбран email-confirmation как альтернатива)
- Onboarding-флоу после регистрации
- Изменение полей регистрационной формы (оставляем текущие 2 шага)
- Access-logic free vs premium (отдельный спринт)
- Восстановление пароля / forgot password (отдельная задача)
- Социальный логин (Google, Apple)

## Зависимости

- **Зависит от:** ничего (первый спринт)
- **Блокирует:** все остальные спринты до 01.07 и после
- **Открытое техническое решение:** custom SMTP провайдер (Resend / SendGrid / Mailgun / Google Workspace SMTP) — решается в процессе

---

## Зафиксированные продуктовые решения

- ✅ **Защита от ботов:** Email-confirmation (НЕ капча)
- ✅ **Поля регистрации:** текущие 2 шага (Шаг 1: имя, фамилия, email, пароль; Шаг 2: kanton, PLZ)
- ✅ **После регистрации:** редирект на главную страницу как залогиненный
- ✅ **Цена Premium:** 19 CHF/мес (для будущих спринтов)

---

## Пользовательские сценарии

### Happy path
- [ ] Юзер открывает модалку регистрации, видит Шаг 1 (имя, фамилия, email, пароль)
- [ ] Заполняет валидные данные → кнопка «Далее» становится активной
- [ ] Кликает «Далее» → видит Шаг 2 (kanton dropdown, PLZ input)
- [ ] Выбирает kanton и вводит PLZ → кнопка «Зарегистрироваться» активна
- [ ] Кликает «Зарегистрироваться» → видит экран «Проверь почту»
- [ ] Получает письмо на указанный email в течение 60 секунд
- [ ] Письмо приходит в Inbox (не в спам)
- [ ] Кликает ссылку в письме → попадает на главную страницу в статусе «залогинен»
- [ ] В Supabase создана запись в `auth.users` с `email_confirmed_at` ≠ null
- [ ] В Supabase создана связанная запись в `profiles` с правильными полями

### Возврат после подтверждения
- [ ] Юзер может закрыть вкладку после клика на ссылку и вернуться — сессия живёт
- [ ] При повторном открытии сайта — залогинен автоматически

### Повторная отправка письма
- [ ] На экране «Проверь почту» есть кнопка «Отправить ещё раз»
- [ ] Кнопка disabled первые 60 секунд после регистрации
- [ ] После 60 секунд кнопка активна → клик → новое письмо приходит
- [ ] Лимит resend: не более 3 раз за час

### Логин до подтверждения
- [ ] Юзер пытается войти с email/password до клика на ссылку
- [ ] Видит сообщение «Подтверди email сначала» с кнопкой «Отправить письмо повторно»
- [ ] Не получает рабочую сессию

## Edge cases

### Валидация полей
- [ ] Невалидный формат email → ошибка под полем «Введи корректный email»
- [ ] Пароль короче 8 символов → ошибка «Минимум 8 символов»
- [ ] Пустое имя или фамилия → ошибка «Поле обязательно»
- [ ] PLZ не цифры или не 4 знака → ошибка «PLZ — 4 цифры»
- [ ] Kanton не выбран → кнопка submit остаётся disabled

### Дубликаты
- [ ] Регистрация с уже существующим подтверждённым email → ошибка «Email уже зарегистрирован»
- [ ] Регистрация с существующим НЕподтверждённым email → resend письма, без новой записи

### Ссылка подтверждения
- [ ] Ссылка протухла (>24ч) → страница «Ссылка устарела» + кнопка «Получить новую»
- [ ] Клик на ссылку повторно после подтверждения → идемпотентно, юзер залогинен, без ошибки
- [ ] Невалидный/обрезанный токен в URL → страница «Ссылка некорректна»

### Сетевые ошибки
- [ ] При offline submit → понятная ошибка «Нет соединения», поля сохранены
- [ ] При 500 от Supabase → ошибка «Что-то пошло не так, попробуй позже»

## UI-инварианты

- [ ] Шаг 1: кнопка «Далее» disabled пока ВСЕ поля не валидны
- [ ] Шаг 2: кнопка «Зарегистрироваться» disabled пока ВСЕ поля не валидны
- [ ] При нажатии submit показывается loading state на кнопке (текст или спиннер)
- [ ] Loading-кнопка не позволяет повторный клик
- [ ] Экран «Проверь почту» — модалка или полный экран (НЕ toast — пользователь должен увидеть инструкцию)
- [ ] Email в подтверждении отображается на экране «Проверь почту» («Мы отправили письмо на user@example.com»)
- [ ] Сообщение содержит контакт поддержки или telegram сообщества на случай проблем
- [ ] Модалки регистрации можно закрыть (×) и переоткрыть без потери введённых данных в течение сессии
- [ ] На экранах ≥320px форма не ломает горизонтальный скролл
- [ ] На мобиле клавиатура не закрывает кнопку submit при фокусе на поле

## Технические инварианты

### Backend
- [ ] При успешной регистрации создаётся запись в `auth.users` И связанная в `profiles` (атомарно или с компенсацией при ошибке)
- [ ] В `profiles` корректно записаны: first_name, last_name, kanton, plz
- [ ] `email_confirmed_at` = null до клика на ссылку
- [ ] `email_confirmed_at` = timestamp после клика
- [ ] Если запись в `profiles` падает — auth-юзер либо откатывается, либо профиль создаётся при первом логине

### SMTP
- [ ] Письмо доставляется в Gmail в Inbox (не в Spam) — тест с реальными адресами
- [ ] Письмо доставляется в Outlook в Inbox (не в Junk)
- [ ] Письмо приходит в течение 60 секунд от submit
- [ ] Отправитель письма — `noreply@slswiss.ch` или подобный (НЕ дефолтный supabase.io)
- [ ] DKIM/SPF/DMARC корректно настроены для отправляющего домена

### Защита
- [ ] Лимит: не более 5 регистраций с одного IP за час
- [ ] Лимит resend письма: не более 3 раз за час на email
- [ ] Сессия не создаётся до подтверждения email

### Чистота кода
- [ ] При стандартном flow в `console.error` ничего не пишется
- [ ] Исправлен дубликат `#login-email-input` (упомянут в HANDOFF.md)
- [ ] Существующие проходящие Playwright-тесты не сломаны

## Решения и заметки

- **2026-06-21 (Planner):** выбран email-confirmation вместо капчи. Это требует фикса SMTP. Скорее всего проблема в дефолтном Supabase SMTP — у него низкая deliverability и Gmail кидает в спам. Нужно подключить custom SMTP. Кандидаты: Resend (бесплатный тир 3000 писем/мес, простая интеграция), SendGrid, или Google Workspace SMTP (у Ксении уже есть для всех 4 доменов).
- **2026-06-21 (Planner):** поля оставляем как есть в HTML — не трогаем форму, только backend.
- **2026-06-21 (Planner):** после регистрации редирект на главную. Не делаем onboarding в этом спринте.
