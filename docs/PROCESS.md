## Роли

| Роль | Где живёт | Когда работает | Что делает |
|---|---|---|---|
| **Planner** | отдельный чат, короткий | раз в неделю, ~15 мин | выбирает спринт из BACKLOG, пишет черновик контракта, финализирует после критики |
| **Agent** | архитектурный чат (длинный) | в середине спринта | строит фичу по утверждённому контракту в `soiludi_v4.html`, пушит |
| **Evaluator** | тестовый чат (длинный) | критика + после пуша | критикует черновик контракта, пишет Playwright из финального контракта, разбирает падения CI |

## Где живут файлы

Всё в Project knowledge:
- `BACKLOG.md` — список спринтов
- `PROCESS.md` — этот файл
- `contracts/<feature>.md` — контракт на спринт
- `slswiss-architecture.md` — техническая архитектура (уже есть)

Когда файл меняется: Claude отдаёт обновлённый `.md` → Kseniia загружает в Project knowledge.

## Цикл одного спринта

### Шаг 1 — Planner открывает спринт
- Читает `BACKLOG.md`
- Берёт верхний пункт
- Пишет черновик `contracts/<feature>.md` по шаблону
- Отдаёт `.md` файл

### Шаг 2 — Evaluator критикует контракт
- Читает черновик
- Добавляет: edge cases, UI-инварианты, mobile, accessibility, console-errors
- Возвращает обновлённый `.md`

### Шаг 3 — Planner финализирует
- Если evaluator добавил слишком много — режет до реалистичного объёма
- Сохраняет финальный контракт в Project knowledge
- В `BACKLOG.md`: спринт → `in_progress`

### Шаг 4 — Agent строит
- Читает финальный контракт
- Реализует в `soiludi_v4.html`
- Коммитит и пушит

### Шаг 5 — Evaluator проверяет
- Пишет Playwright-тесты из пунктов контракта (имя теста = пункт)
- Прогон CI → отчёт сравнивается с контрактом

### Шаг 6 — Петля до зелёного
- Падение CI → site_bug → в architecture chat
- Падение CI → test_bug → в testing chat
- Все пункты контракта зелёные → спринт closed

### Шаг 7 — Закрытие
- `BACKLOG.md`: спринт → `done` + дата
- Если всплыли новые задачи → новые пункты в BACKLOG
- Обновить `slswiss-architecture.md` если архитектура изменилась

## Передача между чатами

Чаты не помнят друг друга. Передача = файл в Project knowledge + короткая выжимка в сообщении.

**Planner → Evaluator** (вставить в начало тестового чата):
> Спринт: `<название>`
> Черновик контракта в Project knowledge: `contracts/<feature>.md`
> Прошу критику: edge cases, UI-инварианты, мобильные сценарии, что я мог пропустить.

**Planner → Agent** (вставить в начало архитектурного чата):
> Спринт: `<название>`
> Финальный контракт: `contracts/<feature>.md`
> Реализуй по пунктам. После пуша — отдай CI-отчёт в тестовый чат.

**Evaluator → Agent** (при падении):
> Спринт: `<название>`
> CI упал на пунктах: `<номер>` — site_bug, строка `<N>`, ошибка `<msg>`
> Фикс → push → пинг.

## Принципы

- Контракт пишется **до** кода и **до** тестов
- Каждый пункт = одно проверяемое утверждение (клик, ассерт, наблюдаемый UI)
- Не можешь проверить пункт автоматически → переформулировать или выкинуть
- 20-40 пунктов на спринт — норма. 60+ — спринт слишком большой, дробить
- Архитектурный документ обновляется **после** спринта, не до
- Контракт неприкосновенен после финализации — изменения = новый спринт

---

## Что считается «готово»

Спринт closed, когда:
1. Все пункты контракта зелёные в CI
2. Финальный коммит в `main`
3. `BACKLOG.md` обновлён
4. Если архитектура поменялась — `slswiss-architecture.md` обновлён
