## ⚡ Конкретные подзадачи (по порядку)

### 1. Инспекция форм (30 мин)

- Открой `soiludi_v4.html`, найди форму каталога и форму событий
- Выпиши РЕАЛЬНЫЕ поля каждой: name/id, тип, обязательность, лимиты
- Найди функции submit обеих форм (сейчас они показывают заглушку)
- Проверь: верхняя кнопка «+ Добавить в каталог» в баннере сломана (из HANDOFF.md) — реши чинить или убрать

### 2. Схема Supabase (1ч)

Создай две таблицы:

**`catalog_submissions`**
- `id` uuid pk default gen_random_uuid()
- `created_at` timestamptz default now()
- `status` text default 'pending_moderation'
- + поля бизнеса из реальной формы

**`event_submissions`**
- `id` uuid pk
- `created_at` timestamptz default now()
- `status` text default 'pending_moderation'
- + поля события из реальной формы

**RLS на обеих (критично — пишем от анонимной роли):**
- Включить RLS
- Политика: `anon` роль может только INSERT
- INSERT принудительно ставит `status = 'pending_moderation'` (аноним не может вставить 'approved')
- НЕ давать anon права SELECT/UPDATE/DELETE

### 3. Wiring формы каталога (1–2ч)

- Замени заглушку submit на реальную вставку в `catalog_submissions`
- Loading state на кнопке во время запроса
- Success-экран показывается ТОЛЬКО после реального ответа Supabase (не ложный)
- При ошибке — НЕ показывать success, показать понятную ошибку
- Двойной клик не создаёт две записи

### 4. Wiring формы событий (1–2ч)

- То же для `event_submissions`
- Дата события в прошлом → ошибка (если применимо)

### 5. Edge cases (1ч)

Из контракта:
- Потеря соединения → ошибка, данные не теряются
- Supabase SDK null при init → retry loop (паттерн initSupabase из архитектуры)
- Длинный ввод, спецсимволы, эмодзи → корректно
- 500/RLS ошибка → честное сообщение, не ложный success

### 6. Проверка (30 мин)

- Отправь реальную заявку каталога → проверь строку в Supabase Dashboard
- Отправь реальное событие → проверь строку
- Убедись что `status = 'pending_moderation'`
- `node --check` на главном script
- `console.error` пустой при нормальном flow
